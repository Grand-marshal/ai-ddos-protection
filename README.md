Ты — ведущий инженер DevSecOps и эксперт по машинному обучению в сфере кибербезопасности. Нам нужно с нуля спроектировать и реализовать MVP-проект: "Интеллектуальная система защиты от DDoS-атак на базе модели Random Forest".

Предыдущая итерация проекта провалилась, потому что модель переобучилась на абсолютные значения объема трафика (метрику пакетов в секунду - PPS). Когда мы проводили демонстрационную слабую атаку с одной виртуальной машины, ML-модель считала ее легитимным трафиком, так как объемы были несопоставимы с полноценным датасетом. Нам запрещено использовать статические правила (rate-limiting, регулярные выражения). Блокировка должна происходить СТРОГО на основе предсказаний ML-модели (вероятность атаки >= 0.9).

### 1. Наше окружение и ограничения
*   **Стенд:** Две виртуалки Kali Linux в VirtualBox на хосте Windows, соединенные через Сетевой мост (Bridged Adapter). Они видят друг друга по локальной сети.
*   **Атакующий:** 1 виртуалка Kali. Ресурсы ограничены (процессор/память хоста). Масштабного ботнета нет, но атака должна симулировать распределенность (например, через спуфинг IP-адресов средствами hping3/Scapy).
*   **Защитник:** 1 виртуалка Kali, на которой в Docker-контейнерах развернуты целевой сайт, ML-модуль и подсистема защиты.

### 2. Стратегия обучения ML (Решение проблемы слабой атаки)
Вместо абсолютных счетчиков (PPS, Bytes/sec), модель должна обучаться на ОТНОСИТЕЛЬНЫХ и СТРУКТУРНЫХ признаках, которые не зависят от мощности атаки. Твоя задача — спроектировать генерацию фич так, чтобы учитывались:
*   Соотношение пакетов (например, Ratio = SYN-пакеты / ACK-пакеты).
*   Распределение длин пакетов (при HTTP-флуде или SYN-флуде они специфичны).
*   Интервалы между пакетами (IAT — Inter-Arrival Time) и их стандартное отклонение.
*   Базовый чистый трафик берем из файла `Benign.csv` (датасет CICD DDoS 2019).

### ТРЕБУЕМАЯ АРХИТЕКТУРА ПРОЕКТА (Реализуем пошагово)

Тебе необходимо спроектировать и написать код для 6 взаимосвязанных компонентов:

#### 1. Модуль Атаки (Kali-Атакующий)
Скрипт (Python/Scapy или bash-обертка над hping3), который генерирует атаку (SYN-флуд и HTTP GET флуд). 
*   *Требование:* Атака должна использовать спуфинг IP-адресов (множество фейковых source IP), чтобы симулировать DDoS с одного устройства. Она должна отличаться по структуре от `Benign.csv` и гарантированно «класть» незащищенный демонстрационный сайт (см. пункт 5), не перегружая при этом саму виртуалку хоста.

#### 2. Конвейер Данных и ML-Модель (Random Forest)
*   Скрипт для создания обучающей выборки: берем `Benign.csv`, собираем pcap нашей атаки из Пункта 1, конвертируем pcap в аналогичный формат признаков (через утилиту типа CICFlowMeter или собственный легковесный парсер на Scapy).
*   Скрипт обучения: обучение классификатора Random Forest. Модель должна выдавать `predict_proba`. Порог срабатывания защиты: вероятность атаки >= 0.9.

#### 3. Приложение Мониторинга (Сниффер трафика)
Python-скрипт, который в реальном времени (или скользящим окном в 1-2 секунды) перехватывает сетевой трафик на защищаемом интерфейсе, агрегирует его в те же фичи, подсовывает обученной ML-модели и отправляет метки (Норма / Атака + Вероятность + IP-источника) в дашборд.

#### 4. Приложение Активной Защиты (Mitigation Engine)
Скрипт, который слушает вердикты от ML-модели. Если модель выдает вероятность атаки >= 0.9 для конкретного IP (или пула IP с аномальным поведением), скрипт динамически применяет правило блокировки (через `iptables` / `nftables`).

#### 5. Демонстрационный Сайт и Дашборд
*   **Сайт:** Максимально простой и ресурсоемкий (например, веб-приложение на Flask/Node.js с искусственно заниженным числом воркеров/потоков, чтобы атака из пункта 1 его гарантированно «дропала» за несколько секунд, если защита отключена).
*   **Дашборд:** Веб-страница (Streamlit / Flask + Chart.js), отображающая в реальном времени график текущего трафика, вердикты ИИ (Атака/Норма) и список заблокированных IP.

#### 6. Docker-окружение
Весь защитный комплекс (Сайт, Дашборд, Сниффер, Модуль защиты) должен быть обернут в Docker-контейнеры.
*   *Важно:* Учти, что контейнеру защиты понадобятся права `--cap-add=NET_ADMIN` и сеть `--network host`, чтобы он мог управлять правилами `iptables` реального сетевого интерфейса Kali-защитника.
